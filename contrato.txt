Contrato de operación: Crear Nuevo Torneo
Actor: Organizador

Operación: registrarTorneo(nombreTorneo, deporteTorneo, categoriaTorneo, formatoTorneo, maxEquipos, fechaInicio, fechaFin)
Referencia Cruzada: C.U. Crear Nuevo Torneo.
Responsabilidades: Validar los datos ingresados del lado del cliente y registrar un nuevo torneo deportivo en el sistema.
Excepciones:
Si el campo nombreTorneo, fechaInicio o fechaFin está vacío, el navegador bloquea el envío y muestra la advertencia "Completa este campo".
Si la fechaFin es anterior a la fechaInicio, el sistema muestra una alerta indicando "La fecha de finalización no puede ser anterior a la fecha de inicio." y cancela la operación.
Si el formatoTorneo es "Fase de Grupos" y maxEquipos es menor a 4, el sistema muestra una alerta indicando "Para el formato 'Fase de Grupos' la capacidad máxima debe ser de al menos 4 equipos." y cancela la operación.
Si el formatoTorneo es "Eliminatoria" y maxEquipos no es una potencia de 2 (ej. 4, 8, 16...), el sistema solicita confirmación sobre la asignación de "Byes" (pases directos) a algunos equipos en la primera ronda. Si el usuario selecciona "Cancelar", se detiene la operation.
Si ocurre un error de comunicación de red al enviar la solicitud al servidor, el sistema muestra la alerta "No se pudo conectar con el servidor." y cancela la operación.
Si el servidor retorna un error de validación (ej. fecha de inicio anterior a la actual), el sistema muestra la alerta "Error: [mensaje]" y cancela la operación.
Pre-condiciones:
El usuario Organizador se encuentra autenticado en su sesión.
Post-condiciones:  
Se creó una instancia de Torneo.
Se asociaron a la instancia de Torneo los datos provistos en los parámetros: nombre_torneo, deporte_torneo, categoria_torneo, formato_torneo, max_equipos, torneo_inicio y torneo_fin.
Se guardó la instancia de Torneo en el sistema.
Se asignó el estado inicial "Pendiente" al torneo creado.
Se mostró la confirmación visual de éxito ("¡Torneo creado con éxito!").
El sistema redirigió al usuario a la vista de administración general de torneos.

Contrato de operación: Crear Nuevo Equipo
Actor: Capitán de Equipo

Operación: registrarEquipo(idEquipo, nombreEquipo, descripcionEquipo, categoriaEquipo, deporteEquipo, torneoAsignado)
Referencia Cruzada: C.U. Crear Nuevo Equipo.
Responsabilidades: Validar los datos ingresados del lado del cliente, registrar un nuevo equipo en el sistema y, si corresponde, solicitar la inscripción al torneo seleccionado.
Excepciones:
Si el campo nombreEquipo o descripcionEquipo está vacío, el navegador bloquea el envío y muestra la advertencia "Completa este campo" o el sistema muestra un mensaje indicando "Todos los campos obligatorios deben ser completados" y cancela la operación.
Si ocurre un error al registrar el equipo (tanto localmente en localStorage como en el backend), el sistema muestra la alerta "Error al registrar el equipo." y cancela la operación.
Pre-condiciones:
El usuario Capitán se encuentra autenticado en su sesión.
Post-condiciones:  
Se creó una instancia de Equipo.
Se asociaron a la instancia de Equipo los datos provistos en los parámetros: id, nombre_equipo, descripcion_equipo, categoria_equipo, deporte_equipo y el correo electrónico del capitán creador.
Se guardó la instancia de Equipo en el sistema (en el almacenamiento de datos local del cliente y se envió al servidor mediante la llamada a la API correspondiente).
Si se seleccionó un torneo en el parámetro torneoAsignado, se creó una instancia de solicitud de inscripción al torneo asociado con el estado inicial "Pendiente".
Se mostró la confirmación visual de éxito ("¡Equipo registrado con éxito! Redirigiendo a gestión de plantilla...").
El sistema redirigió al usuario a la vista de gestión de plantilla del equipo creado.

Contrato de operación: Generar Fixture
Actor: Organizador

Operación: generarFixture(idTorneo)
Referencia Cruzada: C.U. Generar Fixture.
Responsabilidades: Validar los requisitos de participantes del torneo, eliminar partidos previos y generar un nuevo fixture de partidos de forma automática según el formato deportivo seleccionado (Liga, Eliminatoria o Fase de Grupos).
Excepciones:
Si hay cambios pendientes de asignación de equipos participantes (hayCambios === true), el sistema muestra la alerta "Tienes cambios sin guardar en los participantes. Guárdalos primero antes de generar el fixture." y cancela la operación.
Si el número de equipos asignados es menor a 2, el sistema muestra la alerta "Necesitas al menos 2 equipos para generar un fixture." y cancela la operación.
Si el formato es "Fase de Grupos" y la cantidad de equipos asignados es menor a 4, el sistema muestra la alerta "Para el formato 'Fase de Grupos' necesitas registrar al menos 4 equipos." y cancela la operación.
Si el formato es "Eliminatoria" y la cantidad de equipos no es potencia de 2, el sistema muestra un cuadro de confirmación advirtiendo sobre la asignación automática de "Byes" (pases directos) a la siguiente ronda. Si el usuario selecciona "Cancelar", se detiene la operación.
Si el formato es "Liga" y la cantidad de equipos es impar, el sistema muestra un cuadro de confirmación indicando que en cada jornada un equipo quedará libre. Si el usuario selecciona "Cancelar", se detiene la operación.
Si ya existe un fixture previo generado y el usuario cancela la confirmación sobre la eliminación de partidos anteriores, se detiene la operación.
Si ocurre un error de comunicación de red al enviar la solicitud al servidor, el sistema muestra la alerta "Error al conectar con el servidor para generar fixture." y cancela la operación.
Si el servidor/backend retorna un error, el sistema muestra la alerta "Error: [mensaje del servidor]" y cancela la operación.
Pre-condiciones:
El usuario Organizador se encuentra autenticado en su sesión y ha accedido a la pestaña de gestión de fixture del torneo correspondiente.
Post-condiciones:  
Se eliminaron todos los partidos anteriores asignados a este torneo de la base de datos y se limpió el fixture previo del almacenamiento local (localStorage).
Se crearon las nuevas instancias de Partido emparejando a los equipos según el formato del torneo.
Se guardaron las nuevas instancias de Partido en el sistema (base de datos y almacenamiento del cliente).
Se mostró la confirmación visual de éxito ("Fixture generado correctamente bajo el formato: [Formato]").
El sistema actualizó el estado de la aplicación y refrescó la pantalla con el nuevo cronograma y tabla de posiciones.

Contrato de operación: Inscribir Equipo
Actor: Capitán de Equipo

Operación: solicitarInscripcion(idEquipo, idTorneo)
Referencia Cruzada: C.U. Inscribir Equipo.
Responsabilidades: Registrar la postulación de un equipo de la plantilla para participar en un torneo abierto/público disponible.
Excepciones:
Si no se ha seleccionado o registrado un equipo activo, el sistema muestra la alerta "Debes registrar o seleccionar un equipo primero." y cancela la operación.
Si el deporte del equipo no coincide con el deporte del torneo, el sistema muestra la alerta "El deporte del equipo (X) no coincide con el del torneo (Y)" y cancela la operación.
Si la cantidad de jugadores en la plantilla del equipo es inferior al mínimo requerido para el deporte del torneo (Fútbol: 7, Básquet: 5, Vóley: 6, Ping-Pong: 1, Otros: 5), el sistema muestra la alerta "No cumples con el mínimo de jugadores: '[Equipo]' tiene W pero [Deporte] requiere al menos Z jugadores." y cancela la operación.
Si el equipo ya posee una solicitud activa o vinculación registrada con este torneo, el sistema muestra la alerta "Ya has solicitado la inscripción de '[Equipo]' en este torneo." y cancela la operación.
Si el torneo posee cupos libres iguales a cero, el botón de inscripción se encuentra deshabilitado en pantalla con el texto "Cupos Agotados" y no se permite realizar la operación.
Pre-condiciones:
El usuario Capitán se encuentra autenticado en su sesión.
Post-condiciones:  
Se creó una instancia de Solicitud de inscripción en estado "Pendiente".
Se asociaron los datos provistos en los parámetros idEquipo e idTorneo a la solicitud de inscripción creada.
Se persistió la solicitud de inscripción en el sistema (en el almacenamiento del cliente localStorage bajo la clave 'olympia_solicitudes' y se envió la correspondiente petición de persistencia remota al backend).
Se mostró la confirmación visual de éxito ("¡Solicitud enviada para [Equipo]! El organizador la revisará.").
El botón de inscripción cambió su estado en pantalla a inactivo de lectura indicando "SOLICITUD: PENDIENTE".

Contrato de operación: Asignar Asistente
Actor: Organizador

Operación: generarPinAsistente(idTorneo)
Referencia Cruzada: C.U. Asignar Asistente.
Responsabilidades: Generar de manera aleatoria un PIN de acceso alfanumérico temporal único de 6 caracteres para permitir la asignación y el inicio de sesión de un colaborador/asistente de campo sin necesidad de registro formal.
Excepciones:
Si ya existe un PIN activo y el organizador cancela la advertencia de confirmación sobre la invalidación de sesiones previas, se detiene la operación.
Si el navegador bloquea el acceso al portapapeles o no lo soporta, el sistema despliega una alerta indicando que no se pudo copiar automáticamente, permitiendo al usuario seleccionar el PIN en pantalla para copiarlo de manera manual.
Pre-condiciones:
El usuario Organizador se encuentra autenticado en su sesión y ha ingresado a la pestaña "PIN Asistente" del Centro de Control de su torneo.
Post-condiciones:  
Se generó un PIN alfanumérico único de 6 caracteres.
Se guardó el nuevo código PIN en el sistema asociado al identificador del torneo (en localStorage bajo la clave 'olympia_pin_[idTorneo]').
Se invalidaron de forma inmediata todas las sesiones activas previas basadas en códigos anteriores para ese torneo.
Se mostró una alerta de confirmación con la clave generada ("¡Código PIN generado para asistentes: [PIN]!").
Al presionar copiar, se almacenó el PIN en el portapapeles del sistema e indicó visualmente el estado copiado por 2 segundos.

Contrato de operación: Iniciar Sesión (Asistente)
Actor: Asistente de Campo

Operación: iniciarSesionAsistente(nombre, pin)
Referencia Cruzada: C.U. Iniciar Sesión.
Responsabilidades: Identificar y autenticar temporalmente a un colaborador en el sistema para habilitarle el acceso a la carga de resultados de un torneo particular y auditar sus cambios.
Excepciones:
Si el campo nombre de usuario está vacío o solo contiene espacios, el sistema muestra la alerta "Por favor, ingresa tu Nombre y Apellido para identificarte al modificar resultados." y cancela la operación.
Si el campo pin de acceso está vacío, el sistema muestra la alerta "Por favor, ingresa el código PIN de acceso." y cancela la operación.
Si el código PIN ingresado no coincide con ningún PIN activo de torneo en el sistema (excepto por el PIN de fallback 'ref123' utilizado para la demo), el sistema muestra la alerta "El código PIN ingresado es incorrecto o ya ha expirado. Por favor, solicita uno nuevo al organizador." y cancela la operación.
Pre-condiciones:
Debe existir un PIN activo previamente generado para el torneo al que se desea acceder (o utilizar el PIN demo 'ref123').
Post-condiciones:  
Se inició la sesión temporal en el cliente y se registraron en el almacenamiento de sesión del navegador (localStorage) las variables: token activo ('olympia_token' = 'true'), rol ('olympia_role' = 'Asistente'), identificador de torneo asociado ('olympia_asistente_torneo') y nombre del asistente ingresado ('olympia_asistente_nombre') para fines de auditoría.
El sistema redirigió al asistente a la interfaz de carga de resultados de su torneo respectivo (/asistente/[idTorneo]).
